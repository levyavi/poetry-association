# Task 006: Admin Manual Add, Edit, and Delete with Embedding Regeneration

## 0. Complexity Assessment
- Level: High
- Justification:
  - Technical complexity: Three separate write paths (add, edit, delete) each touching text cleaning, duplicate detection, embedding generation, DB writes, and the search cache.
  - Integration complexity: Must invalidate the `SearchService` cache on every successful mutation so search stays consistent.
  - Risk level: Medium-high — mutations are irreversible; a failed embedding must not leave a half-saved poem.
  - Estimated effort: ~450 changed lines.
- Complexity rubric alignment:
  - High: cross-cutting concerns, filesystem + DB interaction, concurrency, or irreversible operations ← fits here (irreversible DB writes + embeddings + cache invalidation).
- Recommended execution mode:
  - High → incremental implementation with validation checkpoints (add → edit → delete).

---

## 1. Goal
Give authenticated admins the ability to create, edit, and delete poems through the admin UI, enforcing the duplicate-cleaned-text rule and regenerating embeddings on every add and on every edit that changes title or body.

---

## 2. Why this task exists now
Task 005 exposed the admin dashboard but the Edit and Delete columns are inert and there is no add form. Delivering CRUD unblocks the full manual workflow, while CSV import (Task 007) and rebuild (Task 008) rely on `update_poem`, `delete_poem`, and the cache-invalidation hook landing here.

---

## 3. Prerequisites
- Task 002: text cleaning, repository, embedding service.
- Task 003: `SearchService` with a `refresh` hook.
- Task 005: admin auth, blueprint, dashboard template with Edit and Delete placeholders.

---

## 4. Scope
- Add form at `GET /admin/poems/new` and `POST /admin/poems`.
- Edit form at `GET /admin/poems/<int:id>/edit` and `POST /admin/poems/<int:id>/edit`.
- Delete action at `POST /admin/poems/<int:id>/delete` with an explicit confirmation step (either a confirmation page or a confirm modal; pick the confirmation page for simplicity and no new JS).
- `repository.update_poem(conn, id, title, text, embedding_service)` that performs text cleaning, duplicate detection (excluding the current row), regenerates the embedding if title or body changed, updates `updated_at`, and writes in a single transaction.
- `repository.delete_poem(conn, id)` that deletes the row and returns a boolean.
- Cache invalidation: every successful mutation calls `SearchService.refresh()` so subsequent searches reflect the change.
- Flash messages: `Saved`, `Deleted`, or detailed error text on failure.
- Template updates: new `admin/add.html`, `admin/edit.html`, `admin/delete_confirm.html`; dashboard gains links for `Add poem`, `Edit`, and `Delete`.
- CSRF: use a minimal per-session CSRF token stored in the Flask session and validated on every POST mutation. This is local-only code, but design doc §13 expects basic security, and adding the token is trivial and prevents accidental cross-site submissions.

---

## 5. Out of Scope
- CSV import — Task 007.
- Rebuild — Task 008.

## 6. Observable Outcome
- Authenticated admin clicks `Add poem`, fills the form, and submits; `Saved` flashes and the dashboard shows the new poem. Searching for a term in the poem retrieves it.
- Admin clicks `Edit` on an existing poem, changes the title, and submits; `Saved` flashes, the dashboard reflects the new title, and `updated_at` advances. The search cache reflects the updated embedding.
- Admin clicks `Delete` on a poem, lands on a confirmation page, confirms, and is redirected to the dashboard with a `Deleted` flash; the poem no longer appears in search.
- Attempting to add or edit a poem whose cleaned text duplicates another poem shows a detailed error and does not mutate the DB.

## 11. Public Functions / Methods
### repository.update_poem
- Purpose: Update a poem's title and text, enforcing duplicate rules and regenerating the embedding when content changes.
- Parameters: `conn`, `id`, `title`, `text`, `embedding_service`.
- Returns: updated `sqlite3.Row`.
- Side effects: DB write inside a transaction.
- Errors: `PoemNotFoundError` if the id does not exist; `DuplicatePoemError` if another row has the same cleaned text; propagates embedding errors.

### repository.delete_poem
- Purpose: Delete a poem by id.
- Parameters: `conn`, `id`.
- Returns: `True` if a row was deleted, `False` if no such id.
- Side effects: DB delete inside a transaction.
- Errors: none beyond SQLite errors.

### csrf.issue_token
- Purpose: Generate and store a per-session CSRF token if not already present.
- Parameters: none.
- Returns: token string.

### csrf.verify_token
- Purpose: Compare a form-submitted token to the session token using `hmac.compare_digest`.
- Parameters: `submitted: str`.
- Returns: `bool`.

### admin.add_poem (GET, POST)
- Purpose: Render and handle the add form.
- Returns: rendered form or redirect to dashboard on success.
- Errors: flashes a detailed message on duplicates or empty text.

### admin.edit_poem (GET, POST)
- Purpose: Render and handle the edit form for an existing poem.
- Returns: rendered form or redirect to dashboard on success.

### admin.delete_poem_confirm (GET)
- Purpose: Render a small confirmation page for the poem about to be deleted.

### admin.delete_poem (POST)
- Purpose: Perform the deletion after confirmation.

## 15. Detailed Implementation Notes
1. CSRF: `csrf.issue_token` stores `session["csrf_token"]` on first request and returns it. Templates call `{{ csrf_token() }}` via a Jinja global registered in `create_app`. Every mutation POST calls `csrf.verify_token(request.form.get("csrf_token", ""))` and aborts 400 on failure.
2. Add flow: GET renders `add.html` with an empty form; POST validates CSRF, strips/validates text, calls `repository.create_poem`, catches `DuplicatePoemError`, re-renders on error with the user's draft preserved, calls `SearchService.refresh()` on success, flashes `Saved`, redirects to the dashboard.
3. Edit flow: GET fetches the existing row via `repository.get_poem`, 404 if missing, renders `edit.html` pre-populated. POST validates CSRF, reads new values, calls `repository.update_poem`, handles errors the same way as add.
4. Inside `update_poem`: fetch existing row, compute new cleaned text / dedup key, check `find_by_cleaned_text(new_key)` — if the found row's id differs from the target id, raise `DuplicatePoemError`. If the stored row's `(title, text)` matches new values exactly, skip embedding re-generation and only bump `updated_at` — otherwise encode + store. This honors the rule that title changes do trigger regeneration.
5. Delete flow: GET `delete_confirm` renders the confirmation screen. POST validates CSRF, calls `repository.delete_poem`, refreshes the search cache, flashes `Deleted`, redirects to dashboard.
6. All mutation routes are wrapped in `login_required`.
7. Dashboard template gains per-row action links/forms: `Edit` is an `<a>` to `/admin/poems/<id>/edit`; `Delete` is a `<form method="GET" action="/admin/poems/<id>/delete">` landing on the confirm page (the actual delete POST lives on the confirm page's form).
8. Flash messages render via the shared `_base.html` flash region.

## 16. Acceptance Criteria
1. Authenticated admin can add a poem via the form and see it in the dashboard and in search results.
2. Attempting to add a poem with text identical (after cleaning) to an existing poem fails with a clear error and does not insert anything.
3. Authenticated admin can edit a poem's title and/or body; `updated_at` advances; the embedding is regenerated when content changes.
4. Editing to a duplicate (excluding the row itself) fails with a clear error.
5. Authenticated admin can delete a poem after the confirmation screen; the row is removed and the poem disappears from search results.
6. Every mutation POST fails with 400 when the CSRF token is missing or wrong.
7. Search cache reflects additions, edits, and deletions immediately after the request completes.
8. All tests pass with real SQLite and a real embedding model.

## 18. Failure Handling & Rollback
- Failure scenarios:
  - Embedding generation fails mid-save → the exception bubbles before the INSERT/UPDATE commit, so nothing is written.
  - Duplicate detection hits → raise before any write.
  - CSRF failure → 400 before any write.
  - Delete of a non-existent id → `delete_poem` returns False, flash an error, no state change.
- Partial failure handling: all writes happen inside a single SQLite transaction per request. Errors raise and the transaction rolls back automatically via the `with conn:` context.
- Rollback strategy: SQLite transactional rollback; no bespoke rollback logic.
- Idempotency considerations: add/edit are NOT idempotent (same request twice can succeed both times for add and have no-op second time for edit with identical content). Delete IS idempotent — a second delete of the same id is a no-op.

---

## 19. Risks and Edge Cases
- Risk: Forgetting to invalidate the search cache → addressed by calling `SearchService.refresh()` after every successful mutation and covered by an integration test.
- Edge case: Editing a poem to its own current content — `update_poem` detects this and skips encoding but still bumps `updated_at` (see below) or keeps `updated_at` stable. Decision: only bump `updated_at` when a real change occurred, to avoid noisy timestamps. Covered by `test_updated_at_preserved_on_untouched_rows`.
- Edge case: Poem title with only whitespace → treated as empty string after trimming.
- Edge case: Very long poem — the textarea accepts it; DB has no length limit.
