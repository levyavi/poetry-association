# Task 009: Persist Lemmatized Search Text and Bundle Offline NLP Resources

## 0. Complexity Assessment
- Level: High
- Justification:
  - Technical complexity: Introduces local NLTK resource packaging, deterministic lexical text generation, live SQLite schema migration, and new metadata needed for V2 compatibility.
  - Integration complexity: Touches packaging, config, app startup, DB initialization, repository writes, CSV import, manual rebuild, CLI import, and the existing admin mutation flows.
  - Risk level: High because poem writes must remain all-or-nothing per row, existing V1 databases must migrate safely, and missing NLP assets must fail clearly without silent fallback downloads.
  - Estimated effort: ~450 changed lines.
- Complexity rubric alignment:
  - Low: single component, no persistence, minimal side effects
  - Medium: multiple components and/or persistence involved
  - High: cross-cutting concerns, filesystem + DB interaction, concurrency, or irreversible operations
- Recommended execution mode:
  - High → incremental implementation with validation checkpoints

---

## 1. Goal
Persist V2-compatible `lemmatized_search_text` for every poem using bundled local NLP resources, and make every existing write path regenerate that derived field safely.

---

## 2. Why this task exists now
Every later V2 feature depends on consistent, persisted lexical text. Exact lexical scoring, synonym lookup, and upgrade detection are all unsafe until the application can build the same lemmatized representation on demand for new data and existing databases.

---

## 3. Prerequisites
- Task 002: repository, CSV import, CLI import, and embedding generation exist.
- Task 006: admin add/edit/delete routes already mutate poems.
- Task 007: admin CSV import UI already exists.
- Task 008: full rebuild flow and rebuild lock already exist.

---

## 4. Scope
- Add NLTK as a packaged offline dependency and bundle the required local resource tree with the application.
- Add a lexical processing module responsible for generating the stored `lemmatized_search_text` from title + poem body using a deterministic normalization and lemmatization pipeline.
- Extend DB initialization to migrate existing V1 databases by adding `lemmatized_search_text` and a persistent metadata table for V2 search-index compatibility.
- Modify poem creation, edit, CSV import, CLI import, and manual rebuild flows so they regenerate `cleaned_text`, `lemmatized_search_text`, and embeddings together before each row is committed.
- Record search-index metadata on a successful full rebuild so later tasks can detect whether automatic upgrade work is still required.
- Keep the public search behavior unchanged in this task; no ranking changes yet.

## 18. Failure Handling & Rollback
Required if task modifies or deletes data.

- Failure scenarios:
  - Bundled NLP resources are missing or misconfigured at startup.
  - Lexical data generation fails during add/edit/import/rebuild.
  - Existing database migration encounters an unexpected SQLite error.
- Partial failure handling:
  - Add/edit writes abort before commit; the existing row stays unchanged.
  - CSV import preserves all rows committed before the failing row, matching V1 semantics.
  - Manual rebuild preserves already-processed rows but does not mark metadata current on failure.
- Rollback strategy:
  - Fix the local NLP resource/configuration issue and rerun the failed action.
  - Re-run manual rebuild after a partial failure to bring the full corpus to a current state.
  - Database migration remains idempotent; no destructive rollback path is required.
- Idempotency considerations:
  - `init_db` and schema migration must be safe to run repeatedly.
  - `build_search_text` is deterministic for the same input.
  - Re-running manual rebuild after a failure is safe and expected.

---

## 19. Risks and Edge Cases
- Packaged NLP assets increase repository/package size: keep only the required local resources and document the exact bundled set.
- Index-time lexical normalization drifts from later query-time normalization: use one shared processor module instead of duplicating logic.
- A migrated V1 database may contain rows with empty titles or unusual punctuation: normalize title + body consistently and preserve existing `Untitled` display behavior.
- Manual rebuild may partially refresh the corpus and leave mixed data temporarily: metadata must remain outdated until a full successful rebuild completes.
