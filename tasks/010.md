# Task 010: Automatic Startup Rebuild for Outdated Search Indexes

## 0. Complexity Assessment
- Level: High
- Justification:
  - Technical complexity: Adds background rebuild orchestration, upgrade-state tracking, and availability gating while the app remains serving status pages.
  - Integration complexity: Touches app startup, rebuild logic, metadata, public routes, admin routes, templates, and rebuild lock behavior.
  - Risk level: High because the system must never search against mixed old/new index formats and must surface failure states clearly.
  - Estimated effort: ~400 changed lines.
- Complexity rubric alignment:
  - Low: single component, no persistence, minimal side effects
  - Medium: multiple components and/or persistence involved
  - High: cross-cutting concerns, filesystem + DB interaction, concurrency, or irreversible operations
- Recommended execution mode:
  - High → incremental implementation with validation checkpoints

---

## 1. Goal
Automatically rebuild outdated search data on startup and keep the application in a safe upgrade state until the rebuild succeeds or an explicit retry is run.

---

## 2. Why this task exists now
Task 009 makes V2 data possible, but existing V1 installations still need their corpus rebuilt before lexical search can be trusted. The design document explicitly requires an automatic first-start rebuild with clear gating and failure handling.

---

## 3. Prerequisites
- Task 008: rebuild lock and manual rebuild route exist.
- Task 009: `lemmatized_search_text` and search-index metadata exist and can be regenerated by rebuild.

---

## 4. Scope
- Detect outdated or legacy `search_index_version` at startup.
- Start an automatic full rebuild in the background when the index is outdated so the app can still serve a status page.
- Extend runtime gating so public search and admin write actions remain disabled while the startup rebuild is running.
- Surface a simple upgrade-in-progress status on the public search page and a read-only banner on admin pages.
- Surface a clear failure state when startup rebuild fails, and keep search disabled until the user retries via restart or the existing explicit rebuild action.
- Refresh in-memory search state automatically after a successful startup rebuild.

---

## 5. Out of Scope
- Progress bars or streaming progress updates.
- Partial or poem-scoped automatic rebuilds.
- Background task infrastructure beyond one process-local coordinator thread.
- Automatic rollback to the old V1 index format.

---

## 6. Observable Outcome
- Launching the app against an outdated V1/V2-legacy database immediately shows a public `search data is being upgraded` message and disables search submission.
- The admin area stays reachable for read-only visibility, but write actions remain blocked while the startup rebuild is running.
- After a successful startup rebuild, the app returns to normal behavior without manual intervention.
- If the rebuild fails, the public page shows a clear error state, search stays disabled, and the admin area exposes the existing rebuild action as the recovery path.

## 10. Classes to Implement or Modify
### StartupUpgradeCoordinator
- Responsibility: Determine whether startup rebuild is required, run it once in the background, and expose the current upgrade status to routes/templates.
- Key methods (names only): `begin_if_needed`, `status`, `requires_rebuild`, `run`
- Interactions: Uses `RebuildLock`, `run_rebuild`, metadata helpers, and `SearchService.refresh`.

### UpgradeStatus
- Responsibility: Represent `idle`, `running`, `failed`, and `ready` states plus any user-visible error message.
- Key methods (names only): `is_search_available`, `is_write_available`
- Interactions: Read by public/admin routes and templates.

## 14. State Transitions
Describe system state before and after this task.
- Before:
  - Outdated V1 databases can exist after upgrade and there is no automatic rebuild path.
  - The app could accidentally start serving against mixed-format search data.
- After:
  - Outdated indexes automatically trigger a full rebuild on startup.
  - Public search and admin writes are unavailable while the rebuild is running.
  - On failure, the app remains in a clear failed-upgrade state until retry.
- Invariants (must always hold true):
  - The app must never silently serve search results from an outdated search index.
  - Only one rebuild can run at a time.
  - A successful startup rebuild always refreshes the in-memory search cache before normal search resumes.

## 15. Detailed Implementation Notes
Step-by-step plan.
- Build a startup coordinator that reads `SearchIndexState` during app creation and decides whether a rebuild is required.
- Keep startup rebuild asynchronous so the browser can load a status page while work is running.
- Reuse the existing rebuild lock and manual rebuild logic instead of creating a second rebuild pipeline.
- Expose coordinator status through `app.extensions` and a template helper so both public and admin views can render a consistent message.
- Treat both `running` and `failed` states as search-unavailable states.
- Allow admin read-only visibility during upgrade so the user can see the current state and access the rebuild retry action after failure.
- After success:
  - update metadata to the current `SEARCH_INDEX_VERSION`
  - record the successful rebuild timestamp
  - refresh `SearchService`
  - transition status back to `ready`
- After failure:
  - record the failure message
  - leave the index marked outdated
  - keep search disabled until retry or restart
- Do not introduce a new retry UI; the existing rebuild action and restart path are sufficient for recovery.

Do not include code.

## 16. Acceptance Criteria
1. Starting the app against an outdated database automatically launches a full rebuild without requiring manual admin steps.
2. While the startup rebuild is running, `GET /` shows an upgrade status message and `POST /search` returns a 503 response.
3. While the startup rebuild is running, admin write routes are blocked but admin read-only pages remain accessible.
4. After a successful startup rebuild, `search_index_version` is current, `last_successful_full_rebuild_at` is updated, and normal search resumes automatically.
5. If the startup rebuild fails, the app surfaces a clear local error and keeps search disabled.
6. A manual rebuild retry from the failed state succeeds and restores normal operation.
7. No automatic rebuild runs when the stored search index is already current.

## 18. Failure Handling & Rollback
Required if task modifies or deletes data.

- Failure scenarios:
  - Automatic rebuild fails mid-corpus.
  - The app process exits during the background rebuild.
  - The rebuild lock is already held when startup upgrade begins.
- Partial failure handling:
  - Preserve the existing rebuild semantics for already-processed rows.
  - Keep the stored metadata outdated so the app does not treat the partial rebuild as complete.
  - Expose the failed-upgrade state and block search until recovery.
- Rollback strategy:
  - Retry via restart or the explicit rebuild action.
  - No downgrade path to V1 search format is supported.
- Idempotency considerations:
  - Re-running the startup rebuild is safe.
  - A process restart against still-outdated metadata should start the automatic rebuild again.

---

## 19. Risks and Edge Cases
- Background rebuild thread may finish before the user loads the page: status handling must still converge cleanly to `ready`.
- A user may open multiple tabs during upgrade: every tab must see the same blocked search state.
- A failed upgrade must not leave search partially available: treat `failed` as unavailable until retry.
- Existing admin read-only views must not accidentally trigger write code paths while upgrade is running.
