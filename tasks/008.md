# Task 008: Rebuild All Embeddings with Application Lock

## 0. Complexity Assessment
- Level: High
- Justification:
  - Technical complexity: Cross-cutting lock that affects both public search and all admin writes; synchronous bulk re-embedding.
  - Integration complexity: Touches the public search route, every admin mutation (add, edit, delete, CSV preview / confirm / cancel), the search cache, and the embedding service.
  - Risk level: Medium-high — concurrent access during rebuild is the key correctness concern, and rebuild failures leave the DB in a partially-re-embedded state.
  - Estimated effort: ~350 changed lines.
- Complexity rubric alignment:
  - High: cross-cutting concerns, filesystem + DB interaction, concurrency, or irreversible operations ← fits here (affects every write route + search).
- Recommended execution mode:
  - High → incremental implementation with validation checkpoints (lock module → rebuild loop → route wiring).

---

## 1. Goal
Add a `Rebuild all embeddings` button to the admin dashboard that regenerates every poem's embedding using the current model while blocking public search and all admin write actions for the duration of the rebuild.

---

## 2. Why this task exists now
Tasks 002–007 assume embeddings are consistent with the loaded model. Model updates or corruption recovery require a manual rebuild. This task completes the V1 admin scope and also finalizes concurrency handling that all prior write routes must respect.

---

## 3. Prerequisites
- Task 002: embedding service and repository.
- Task 003: `SearchService` with `refresh`.
- Task 005: admin auth.
- Task 006: admin mutation routes to protect.
- Task 007: admin import routes to protect.

---

## 4. Scope
- `locks.py` module exposing a process-wide `RebuildLock` with `acquire`, `release`, `is_rebuilding`, and a context manager.
- `rebuild.py` module with `run_rebuild(conn, embedding_service)` that walks every poem row, regenerates its embedding, and commits the update.
- `POST /admin/rebuild` route that acquires the lock, runs the rebuild, releases the lock, refreshes the search cache, and renders a result page.
- Lock gate on `POST /search` → return 503 with a simple `Search is temporarily disabled while embeddings are being rebuilt.` message (rendered on the search page so the user sees the reason).
- Lock gate on every admin mutation route (add, edit, delete, CSV preview / confirm / cancel) → redirect back with a flash `Rebuild in progress — write actions are disabled.`.
- Admin dashboard gains a visible `Rebuild all embeddings` button with CSRF-protected form, a simple loading state (browser default spinner), and a post-run flash message.
- Result page showing rebuilt count and any error.

---

## 5. Out of Scope
- Progress streaming or a progress bar.
- Backup / snapshot of pre-rebuild embeddings (explicitly out of V1 per design doc §18.6).
- Background job execution — the rebuild runs synchronously in the request thread.
- Partial-range rebuilds or poem-level rebuild — only full rebuild is supported.

---

## 6. Observable Outcome
- Authenticated admin clicks `Rebuild all embeddings`, confirms, and the browser blocks until the rebuild finishes (seconds for a ~330-poem corpus).
- During the rebuild, any `POST /search` from another tab returns an HTTP 503 with a rendered message explaining the rebuild is in progress.
- During the rebuild, any admin mutation (add / edit / delete / CSV routes) redirects with a flash stating writes are disabled.
- After the rebuild completes, every row's embedding has been regenerated (verifiable by comparing a before/after hash of one row's embedding BLOB), the search cache reflects new embeddings, and the admin sees a `Rebuilt N embeddings` flash.
- If the rebuild raises mid-loop, the lock is released, the admin sees a detailed error, and the DB contains a mix of old and new embeddings (documented in §18).

## 10. Classes to Implement or Modify
### RebuildLock
- Responsibility: Provide a single, process-wide exclusive lock with an `is_rebuilding` probe.
- Key methods: `acquire`, `release`, `is_rebuilding`, `__enter__`, `__exit__`.
- Interactions: Acquired by the rebuild route; probed by search and admin mutation routes.
- Internals: `threading.Lock` plus a boolean flag updated under the lock.

---

## 11. Public Functions / Methods
### RebuildLock.acquire
- Purpose: Non-blocking acquire; returns True on success, False if already held.
- Parameters: none.
- Returns: `bool`.

### RebuildLock.release
- Purpose: Release the lock and clear `is_rebuilding`. Safe to call multiple times.

### RebuildLock.is_rebuilding
- Purpose: Return the current rebuild state.
- Returns: `bool`.

### rebuild.run_rebuild
- Purpose: Iterate every poem, regenerate and store the embedding, return counts.
- Parameters: `conn`, `embedding_service`.
- Returns: `RebuildResult(total, rebuilt, error)`.
- Side effects: DB writes.
- Errors: Propagates the first row-level exception after partial progress is committed.

### admin.rebuild_all (POST)
- Purpose: Acquire the lock, run the rebuild, refresh the search cache, release the lock, render the result page.
- Errors: If the lock is already held, flash `Rebuild already in progress` and redirect back.

### public.search (updated)
- Additional behavior: returns 503 with a rendered search page containing a rebuild-in-progress notice when the lock is held.

### admin.reject_if_rebuilding (helper)
- Purpose: Check the lock and, if held, flash a message and return a redirect. Called as the first line of every admin mutation view.

## 14. State Transitions
- Before: any admin action may run and search is always available.
- After:
  - While a rebuild is in flight, search and admin writes are rejected.
  - After a successful rebuild, every poem's embedding is freshly generated and the search cache is refreshed.
  - After a failed rebuild, some poems have new embeddings and some still have old ones, the lock is released, and the admin sees an error.
- Invariants:
  - Only one rebuild can run at a time.
  - The lock is always released — success or failure — via try/finally.
  - No poem ever ends up with a missing embedding.
  - While the lock is held, no new rows are inserted and no existing rows are mutated outside of the rebuild loop.

---

## 15. Detailed Implementation Notes
1. `RebuildLock` wraps `threading.Lock`. `acquire` tries to acquire with `blocking=False`; on success sets `self._rebuilding = True` under the lock and returns True. On failure returns False.
2. `release` clears the flag and releases the lock.
3. `is_rebuilding` reads the flag (no mutation). Reading an atomic boolean is safe under CPython's GIL for this purpose.
4. `rebuild.run_rebuild` opens a cursor over `SELECT id, title, text FROM poems ORDER BY id`. For each row, encode a fresh embedding, serialize, `UPDATE poems SET embedding = ?, updated_at = ? WHERE id = ?`, commit per-row so partial progress survives a later failure. Count successes; on exception, record `error` and break.
5. Admin rebuild route: `if not lock.acquire(): flash(...); return redirect(...)`. Inside a `try/finally`, open a DB connection, call `run_rebuild`, call `search_service.refresh()`, render the result page. `finally: lock.release()`.
6. Public `/search` checks `lock.is_rebuilding()` at the top; if True, return a rendered `search.html` with a `rebuild_in_progress=True` flag and HTTP 503.
7. `admin.reject_if_rebuilding` returns `None` when idle and a redirect response when rebuilding. Every mutation view starts with `if (resp := reject_if_rebuilding()): return resp`.
8. The admin banner is rendered in `_base.html` via a Jinja global that exposes `rebuild_lock.is_rebuilding()`.
9. Tests use a direct `RebuildLock` instance or the Flask app's lock extension, asserting state transitions without starting a real HTTP server.

## 16. Acceptance Criteria
1. Admin dashboard has a visible `Rebuild all embeddings` button that runs a full rebuild.
2. After a successful rebuild, every poem's embedding has been regenerated (verified by a before/after comparison on at least one row).
3. During a rebuild, `POST /search` returns HTTP 503 and a rendered message.
4. During a rebuild, every admin mutation route (add, edit, delete, CSV preview, CSV confirm, CSV cancel) is rejected with a redirect and a flash.
5. The lock is released after both successful and failed rebuilds.
6. A second rebuild triggered while the first is in progress is rejected with a flash, not a crash.
7. `SearchService.refresh()` is called exactly once after a successful rebuild.
8. All tests pass.

## 18. Failure Handling & Rollback
- Failure scenarios:
  - Rebuild loop fails on row K → rows 1..K-1 already have new embeddings committed; rows K..N still have old embeddings. The lock is released via `finally`. The admin sees an error page. This partial state is acceptable per the V1 scope (backup/restore is explicitly out — design doc §18.6). Admin can retry the rebuild; since embeddings are deterministic, rerunning is safe.
  - Lock acquisition fails (another rebuild in progress) → reject with a flash; no state change.
  - Process crash mid-rebuild → on next startup the DB has mixed embeddings; admin can click rebuild again.
- Partial failure handling: per-row commits.
- Rollback strategy: None (no pre-rebuild snapshot in V1). Re-run is the supported recovery.
- Idempotency considerations: Rebuild is idempotent on a per-row basis because embeddings are deterministic for a fixed model.

---

## 19. Risks and Edge Cases
- Risk: Forgetting to release the lock on an unexpected exception → mitigated by `try/finally` in the rebuild route and tested by `test_lock_released_after_rebuild_failure`.
- Risk: Cascading flashes flooding the session if many admin write attempts happen during a rebuild → each request yields one flash per attempt, acceptable for V1.
- Edge case: DB has 0 poems → `run_rebuild` returns `total=0, rebuilt=0` and the result page shows `Rebuilt 0 embeddings`.
- Edge case: Admin kicks off a rebuild on a different browser tab than the one polling search → the search tab sees 503 until the rebuild tab finishes.
- Risk: CPython GIL scheduling might briefly allow a search request to start before the `is_rebuilding` flag is observed → because the flag is set under the lock before any real work begins, the race window is at most a nanosecond-level slice and is acceptable for a private local tool.
