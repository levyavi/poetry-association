  - Risk level: High because polysemy noise and incorrect POS handling can degrade relevance if the rules are not implemented exactly.
  - Estimated effort: ~390 changed lines.
- Complexity rubric alignment:
  - Low: single component, no persistence, minimal side effects
  - Medium: multiple components and/or persistence involved
  - High: cross-cutting concerns, filesystem + DB interaction, concurrency, or irreversible operations

## 1. Goal
Add conservative noun/adjective synonym expansion to lexical scoring, behind a global config flag, without changing the semantic embedding query or the user-facing UI.

---

## 2. Why this task exists now
Task 011 establishes the combined-scoring framework with exact lexical matches only. Synonym expansion should be added only after that exact path is stable, so synonym behavior is isolated and can be disabled safely if needed.

---

## 3. Prerequisites
- Task 009: lexical processor and offline NLTK resources exist.
- Task 010: upgraded databases are rebuilt before search uses V2 lexical logic.
- Task 011: combined ranking and exact lexical scoring are already in place.

---

## 4. Scope
- Add a synonym expansion module that uses local WordNet only.
- POS-tag query words and restrict synonym expansion to nouns and adjectives.
- Use only the top 1 synset per eligible word.
- Discard multi-word synonyms, empty/invalid terms, stopwords, and the original query word itself.
- Normalize and deduplicate synonym candidates, keeping at most 5 single-word synonyms per eligible query word.
- Apply synonym matches at `0.7` only when an exact lexical match is absent for that query word.
- Add a global `ENABLE_SYNONYM_EXPANSION` config flag defaulting to enabled.
- Keep the semantic embedding query unchanged and keep the public UI implicit.

---

## 5. Out of Scope
- In-memory synonym cache.
- Search diagnostics logging.
- Any admin synonym editor, blacklist, or override system.
- Phrase-level thesaurus expansion, verb expansion, adverb expansion, or negation handling.

---

## 6. Observable Outcome
- A query can now retrieve a poem through synonym-only lexical evidence when semantic similarity is above the floor.
- Disabling `ENABLE_SYNONYM_EXPANSION` removes synonym-based lexical boosts without code changes.
- The public search experience still looks the same: one input, same result list, same modal flow.

## 11. Public Functions / Methods
### LexicalTextProcessor.build_tagged_query_terms
- Purpose:
  - Build normalized lexical query terms plus the POS information needed to decide synonym eligibility.
- Parameters:
  - `query`
- Returns:
  - ordered list of tagged lexical query terms
- Side effects:
  - None
- Errors:
  - If POS tagging fails for a word, that word remains eligible for exact matching but not synonym expansion.

### SynonymExpander.expand_term
- Purpose:
  - Resolve the conservative synonym set for one eligible query word.
- Parameters:
  - `term`
  - `pos_tag`
- Returns:
  - ordered list of normalized synonym terms
- Side effects:
  - Reads local WordNet resources only.
- Errors:
  - On lookup failure, returns no synonyms and allows search to continue.

### SynonymExpander.expand_terms
- Purpose:
  - Resolve synonym sets for all eligible query words in one request-local call.
- Parameters:
  - tagged query terms
- Returns:
  - mapping of normalized query term to normalized synonym list
- Side effects:
  - None beyond local WordNet reads.
- Errors:
  - Individual term failures do not abort the full search request.

### SearchService.search
- Purpose:
  - Apply synonym fallback scoring after exact-match checks, while preserving the hard semantic floor and combined ranking rules from Task 011.
- Parameters:
  - existing search parameters
- Returns:
  - `list[SearchResult]`
- Side effects:
  - None
- Errors:
  - Lookup failures degrade to exact lexical matching only for the affected term.

## 15. Detailed Implementation Notes
Step-by-step plan.
- Add the new config flag to `Config.from_environment`, defaulting to enabled.
- Extend lexical query processing to preserve the normalized term order and POS information needed for expansion.
- Implement `SynonymExpander` to:
  - map NLTK POS tags to WordNet noun/adjective categories only
  - use only the top returned synset
  - filter out multi-word synonyms and invalid tokens
  - normalize synonym candidates through the same lexical normalization pipeline
  - deduplicate after normalization
  - cap the final set at 5 synonyms
- In `SearchService.search`, keep the lexical scoring loop per query word:
  - exact match = `1.0`
  - else synonym match = `0.7` if enabled and the word is eligible
  - else `0.0`
- Preserve the hard semantic floor from Task 011 exactly.
- Do not change templates or route mechanics; this is a ranking-only slice.

## 18. Failure Handling & Rollback
Required if task modifies or deletes data.

- Failure scenarios:
  - POS tagging fails for one query word.
  - WordNet lookup fails for one eligible query word.
  - A bad synonym set would create obvious retrieval noise.
- Partial failure handling:
  - Skip synonym expansion for the affected word and continue search.
  - Preserve exact lexical and semantic scoring for the rest of the query.
- Rollback strategy:
  - Set `ENABLE_SYNONYM_EXPANSION=false` and restart the app to revert to exact lexical + semantic ranking.
- Idempotency considerations:
  - Re-running the same query against the same corpus and config must remain deterministic.

---

## 19. Risks and Edge Cases
- WordNet polysemy can introduce noisy matches: mitigate by top-1-synset only and a hard 5-term cap.
- Query words with punctuation or casing variants may collapse to the same normalized form as synonyms: dedupe after normalization, not before.
- Queries containing only verbs/adverbs still need exact lexical matching even though they get no synonyms.
- The config flag must disable synonym logic cleanly without breaking search result rendering or exact lexical scoring.
