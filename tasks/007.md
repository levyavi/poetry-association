# Task 007: CSV Import in Admin UI with Pre-Confirm Counts and Cancellation

## 0. Complexity Assessment
- Level: High
- Justification:
  - Technical complexity: File upload handling, two-step workflow (preview then confirm), cancellation via a shared cancel flag, partial preservation on failure.
  - Integration complexity: Reuses `csv_import.plan` / `execute` from Task 002 and plugs into the admin blueprint, CSRF, and search cache invalidation.
  - Risk level: Medium-high — bulk mutations are irreversible, and partial-failure semantics must match design doc §6.3.2.
  - Estimated effort: ~500 changed lines.
- Complexity rubric alignment:
  - High: cross-cutting concerns, filesystem + DB interaction, concurrency, or irreversible operations ← fits here (upload, DB writes, cancellation flag shared between threads).
- Recommended execution mode:
  - High → incremental implementation with validation checkpoints (upload → preview → execute → cancel).

---

## 1. Goal
Let authenticated admins upload a CSV file, review counts of importable and duplicate rows, confirm the import (or cancel mid-run), and see the dashboard updated with the newly imported poems while preserving any rows written before a cancellation or a row-level failure.

---

## 2. Why this task exists now
Task 002 landed the CSV import pipeline as a CLI utility. This task surfaces the same pipeline in the admin UI, which is the only way design doc §6.2.2.4 and §6.3 can be satisfied.

- Task 002: `csv_import.plan`, `csv_import.execute`, duplicate detection.

## 4. Scope
- Upload form at `GET /admin/import` (part of the admin blueprint).
- `POST /admin/import/preview`: accept the uploaded CSV, persist it to a temporary file under the configured data directory (outside the DB), run `csv_import.plan`, store the plan metadata (temp path + filename + counts) in the session, render a preview page showing counts and a confirm button.
- `POST /admin/import/confirm`: read the stored plan metadata, open the temp CSV again (or read the parsed plan from a cache), run `csv_import.execute` with a cancellation flag that checks a shared in-process dict keyed by session id, and render a result screen showing imported count, skipped count, cancellation state, and any error message.
- `POST /admin/import/cancel`: toggle the cancellation flag for the current session so the in-flight execute loop stops at the next row boundary.
- Cleanup: delete the temporary CSV file after confirm / cancel / error, and after idle timeout.
- Search cache refresh after any import that wrote at least one row.

---

## 5. Out of Scope
- Row-level validation preview (explicitly out of V1 per design doc §6.3.4).
- Background job runner or streaming progress — the synchronous request loop is acceptable for ~330 poems.
- Rebuild lock integration — Task 008 adds the rebuild lock and this task's import routes will then start honoring it.

---

## 6. Observable Outcome
- Authenticated admin visits `/admin/import`, selects a CSV file, and clicks `Preview`.
- A preview page shows `N rows to import`, `M rows to skip as duplicates`, along with the filename.
- Clicking `Confirm import` runs the import. The result screen shows `Imported N, skipped M duplicates` and a link back to the dashboard. Newly imported poems appear on the dashboard and in search.
- Clicking `Cancel` mid-import (another browser tab, or via a visible button on the progress screen) aborts the import at the next row boundary; previously imported rows remain; the result screen reflects `cancelled=True`.
- If the CSV has wrong headers, preview fails with a detailed error and no temp file is retained.

## 10. Classes to Implement or Modify
### ImportSession
- Responsibility: Bundle of state for one in-progress admin CSV import (plan, cancel flag, temp path, filename, started-at).
- Key methods: `cancel()`, `is_cancelled()`, `cleanup()`.
- Interactions: Stored by `import_state`; consumed by admin import views; `cancel_flag` callable passed into `csv_import.execute`.

---

## 11. Public Functions / Methods
### import_state.create
- Purpose: Create an `ImportSession` tied to the current admin session id, store it, return it.
- Parameters: `session_id`, `plan`, `temp_path`, `filename`.
- Returns: `ImportSession`.

### import_state.get
- Purpose: Fetch the active session for a given key.
- Parameters: `session_id`.
- Returns: `ImportSession` or `None`.

### import_state.cancel
- Purpose: Set the cancel flag on the session.
- Parameters: `session_id`.
- Returns: `bool` (whether a session existed).

### import_state.discard
- Purpose: Remove the session and delete its temp file.
- Parameters: `session_id`.

### admin.import_upload (GET)
- Purpose: Render the upload form.

### admin.import_preview (POST)
- Purpose: Save the uploaded file to a temp path, run `csv_import.plan`, store state, render the preview page.

### admin.import_confirm (POST)
- Purpose: Run `csv_import.execute` with the stored plan and cancel flag, refresh search cache, render the result page, and then discard state.

### admin.import_cancel (POST)
- Purpose: Set the cancel flag. Returns 204 No Content — the in-flight confirm handler will notice on its next iteration.

  - Upload → Preview (valid CSV) or Error (invalid CSV).
  - Preview → Confirm → Result (Success / Partial / Cancelled / Error).
- Error states:
  - Missing or wrong headers.
  - File too large for the OS to write the temp file.
  - No file selected.

## 14. State Transitions
- Before: DB contains N poems. No admin import capability through the UI.
- After:
  - DB may contain N + imported_count poems.
  - Any temp file created during the flow has been deleted.
  - `import_state` has no stale entries for the current session.
- Invariants:
  - Each successfully imported row has a non-null embedding and a unique cleaned dedup key.
  - Partial preservation: rows written before a cancellation or error remain committed (design doc §6.3.2).
  - Temp files always cleaned up, even on error.
  - The search cache is refreshed exactly once per confirm run (if any row was written).

## 15. Detailed Implementation Notes
1. Upload handler validates presence of a file; writes it to `config.import_temp_dir / f"{uuid4()}.csv"`; opens it and runs `csv_import.plan`. On failure, delete the temp file and re-render the upload page with an error.
2. Store `ImportSession(session_id, plan, temp_path, filename)` in `import_state`. Use the Flask session id — derived from `session.sid` if available, or a stable random token stored in the session (simpler).
3. Preview page reads state by session id and renders counts.
4. Confirm handler opens a DB connection, sets `ImportSession.started_at = now`, builds a cancel flag closure `lambda: session.is_cancelled()`, calls `csv_import.execute(conn, plan, embedding_service, cancel_flag=cancel_flag)`, calls `search_service.refresh()` if `result.imported > 0`, deletes the temp file, discards state, renders the result page.
5. Cancel handler looks up the session by id and sets its cancel flag. If no session exists, return 404. The in-flight confirm handler will check the flag before each row and stop gracefully, returning the current counts plus `cancelled=True`.
6. Because the confirm handler and the cancel handler run in separate requests (and Flask's default dev server serves them on separate threads), the cancel flag must be a thread-safe primitive. Use `threading.Event` inside `ImportSession` — `cancel()` sets it, `is_cancelled()` reads it, and the `cancel_flag` closure calls `event.is_set()`.
7. Discard state is idempotent — calling `discard` on a missing session is a no-op.
8. Add a housekeeping function `import_state.cleanup_expired(max_age_seconds=3600)` called at the top of every admin request. This guarantees orphaned temp files and stale sessions are cleaned up even if the admin closes the browser mid-flow.
9. All mutation routes require `login_required` and CSRF verification.
10. The `Cancel` button on the preview page (before confirm) simply discards the session and redirects to the upload screen.

## 16. Acceptance Criteria
1. Admin can upload a valid CSV, see the preview counts, confirm, and land on a success screen.
2. Uploaded CSV with wrong headers is rejected with a clear error and no temp files remain.
3. In-file duplicates and existing-DB duplicates are both reflected in the `duplicates` count.
4. Clicking cancel mid-import stops further rows from being imported, preserves any already-committed rows, and the result page shows `cancelled=True`.
5. A row-level failure mid-import preserves prior rows and reports the error clearly.
6. Temp files are cleaned up after success, cancellation, error, or session expiry.
7. Search cache reflects newly imported poems immediately after the confirm request completes.
8. All import mutation routes require authentication and a valid CSRF token.
9. All tests pass.

## 18. Failure Handling & Rollback
- Failure scenarios:
  - Upload missing → re-render upload page with error.
  - Plan fails (bad headers) → delete temp file, re-render upload page with error.
  - Execute row fails → commit prior rows, delete temp file, render result with error and failing row index.
  - Cancellation → commit prior rows, delete temp file, render result with `cancelled=True`.
  - Orphaned session (admin closes browser) → `cleanup_expired` deletes temp file on next admin request after TTL.
- Partial failure handling: per-row commits via `create_poem`'s transaction ensure partial success survives any error.
- Rollback strategy: No rollback of already-imported rows (explicit per design doc §6.3.2).
- Idempotency considerations: Re-uploading the same CSV is a no-op thanks to duplicate detection.

---

## 19. Risks and Edge Cases
- Risk: Race between `cancel` and the confirm loop — mitigated by `threading.Event` atomic semantics.
- Risk: A second admin tab uploads a different CSV while the first is pending → two `ImportSession` entries under the same session id would collide. Mitigation: the second upload replaces the first (discard the previous state) and prints a warning flash.
- Edge case: CSV with 0 rows (header only) → preview shows `0 to import, 0 duplicates`, confirm is a no-op that renders a result with zero counts.
- Edge case: CSV containing a blank row → silently skipped.
- Edge case: Filename with spaces or non-ASCII → temp file uses a generated UUID, not the uploaded name.
