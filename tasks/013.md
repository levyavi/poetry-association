## 10. Classes to Implement or Modify
### SynonymExpander
- Responsibility:
  - Cache processed synonym artifacts for repeated eligible query words during the process lifetime.
- Key methods (names only):
  - `expand_term`
- Interactions:
  - Used by `SearchService.search`; cache is reset only when the app process restarts.

### SearchService
- Responsibility:
  - Emit structured local diagnostics for each search request without altering user-facing responses.
- Key methods (names only):
  - `search`
- Interactions:
  - Uses the logger configured in the app and data returned by `SynonymExpander`.

---

## 11. Public Functions / Methods
### SynonymExpander.expand_term
- Purpose:
  - Return the processed synonym list for a normalized eligible term, using the process-local cache when available.
- Parameters:
  - `term`
  - `pos_tag`
- Returns:
  - normalized synonym list plus cache-hit/miss information usable by the search logger
- Side effects:
  - Updates the in-memory cache.
- Errors:
  - Cache corruption or lookup failure causes that term to be recomputed or skipped, not a full search failure.

### SearchService.search
- Purpose:
  - Execute V2 ranking and emit structured local diagnostics for the query, expansions, per-result scores, and match reasons.
- Parameters:
  - existing search parameters
- Returns:
  - `list[SearchResult]`
- Side effects:
  - Writes local structured log entries when the configured log level enables them.
- Errors:
  - Logging failures must not break the search response.

## 15. Detailed Implementation Notes
Step-by-step plan.
- Add a log-level config field that defaults to a quiet local setting but can be raised for debugging.
- Extend `SynonymExpander` with a small in-memory cache storing:
  - normalized eligible term
  - resolved POS category used for WordNet
  - final processed synonym list
- Treat cache invalidation as process restart only; do not add persistence or manual eviction UI.
- In `SearchService.search`, log at least:
  - original query
  - normalized semantic query
  - lexical query words after stopword removal
  - POS tags used for synonym eligibility
  - synonym expansions actually used
  - cache hit/miss status
  - per-result semantic, lexical, and final scores
  - per-result match reasons by query word
  - triggering synonym when a synonym match occurs
- Keep logs structured and local; do not log full poem text.
- Build one deterministic regression corpus file and reuse it across end-to-end tests instead of scattering ad hoc fixtures.
- Add acceptance-style tests that prove the final V2 system works end-to-end, including upgrade rebuild and lexical regeneration behavior.

## 18. Failure Handling & Rollback
Required if task modifies or deletes data.

- Failure scenarios:
  - Cache entries become invalid for one term.
  - Logger configuration is invalid or the logger raises during emission.
- Partial failure handling:
  - Drop or recompute the affected cache entry and continue search.
  - Suppress logging failures so the request still succeeds.
- Rollback strategy:
  - Reduce the log level or remove this task if diagnostics prove too noisy.
  - Cache resets automatically on process restart.
- Idempotency considerations:
  - Cache warmup is safe to repeat.
  - Regression tests must pass the same way on repeated runs against the same fixture data.

---

## 19. Risks and Edge Cases
- Overly verbose logs can make debugging harder: keep fields structured and omit poem body text.
- Cache keys must use normalized eligible terms, not raw query tokens, or hit rates will be misleading.
- Config changes require restart because cache and logger state are process-local; that behavior should be documented rather than hidden.
- Deterministic regression tests can become flaky if they rely on uncontrolled embeddings; use the same fixed fixture corpus and established deterministic helpers already present in the test suite where needed.
