## 1. Goal
Stand up a runnable Python Flask web app that initializes a SQLite database with the `poems` schema at startup and serves a minimal search page shell so the project is visibly alive end-to-end.

## 11. Public Functions / Methods
### create_app
- Purpose: Build and return a configured Flask app instance.
- Parameters: optional `config_override: Config` (used by tests).
- Returns: `Flask` instance.
- Side effects: Ensures the SQLite DB file and schema exist by calling `init_db`.
- Errors: Raises `RuntimeError` if the configured DB path is unwritable.

### init_db
- Purpose: Create the `poems` table and supporting index idempotently.
- Parameters: `db_path: str`.
- Returns: `None`.
- Side effects: Writes to the filesystem (creates or opens the SQLite file and runs DDL).
- Errors: Propagates `sqlite3.Error`.

### get_connection
- Purpose: Return a configured `sqlite3.Connection`.
- Parameters: `db_path: str`.
- Returns: `sqlite3.Connection` with row factory set.
- Side effects: Opens the file.
- Errors: Propagates `sqlite3.Error`.

### public.index
- Purpose: Render the search page shell.
- Parameters: none.
- Returns: Rendered `search.html`.
- Side effects: None.
- Errors: None.

## 15. Detailed Implementation Notes
1. Create `pyproject.toml` with a `[project]` section naming `poem_assoc`, Python ≥3.11, and dependencies `Flask`, `pytest` (dev).
2. Build `config.py` first: `Config` dataclass with fields `db_path`, `secret_key`, `admin_password`, and a `from_environment` classmethod reading `POEM_DB_PATH` (default `./poem_assoc.db`), `POEM_SECRET_KEY` (default generated per-process), `POEM_ADMIN_PASSWORD` (default empty string — not used until Task 005).
3. Build `db.py` with `init_db` running the `CREATE TABLE IF NOT EXISTS` statement and `CREATE INDEX IF NOT EXISTS`.
4. Build `app.py` `create_app` that: loads config, calls `init_db`, sets Flask `SECRET_KEY`, registers the public blueprint, returns the app.
5. Build `routes/public.py` with a single `/` route rendering `search.html`.
6. Templates: `base.html` with `<html>`, `<head>` linking `styles.css`, viewport meta tag; `search.html` extends base and contains the form.
7. `styles.css` uses a basic centered flex layout, max width container, readable font stack, responsive rule for narrow screens.
8. `__main__.py` instantiates `create_app()` and calls `app.run(host="127.0.0.1", port=5000)`.
9. Add a printed startup banner to stdout: `"poem_assoc running on http://127.0.0.1:5000"`.
10. pytest `conftest.py` provides a `temp_db_path` fixture using `tmp_path` and an `app` fixture building a Flask app bound to that temp DB, plus a `client` fixture for the test client.

## 18. Failure Handling & Rollback
- Failure scenarios:
  - DB path is unwritable → `init_db` raises and `create_app` propagates, the process exits with a readable traceback.
  - Port 5000 already in use → Flask raises; user re-runs with an alternative port via standard Flask flags.
- Partial failure handling: `init_db` runs all DDL in a single connection so failure leaves no half-created schema (SQLite auto-rolls back on error).
- Rollback strategy: Deleting the DB file restores the pre-task state since no real data is stored yet.
- Idempotency considerations: All `CREATE` statements use `IF NOT EXISTS`; re-running the app must be safe.

---

## 19. Risks and Edge Cases
- Risk: User runs the app from a directory without write permission → mitigated by a clear startup error.
- Risk: Python package not installed in editable mode → `pyproject.toml` documents `pip install -e .` in the README.
- Edge case: Existing `poem_assoc.db` from a future task already contains rows → `IF NOT EXISTS` guarantees this task never destroys data.
