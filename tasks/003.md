# Task 003: Search Pipeline and Results Rendering with Relevance Labels

## 0. Complexity Assessment
- Level: Medium
- Justification:
  - Technical complexity: Cosine ranking over in-memory embeddings, label thresholds, tie-breaking, Flask route, template rendering.
  - Integration complexity: Touches the repository, embedding service, public route, and the search template — all existing from prior tasks.
  - Risk level: Low — no destructive operations; changes are additive and read-only.
  - Estimated effort: ~350 changed lines.
- Complexity rubric alignment:
  - Medium: multiple components and/or persistence involved ← fits here (repository read + embedding + routing + template).
- Recommended execution mode:
  - Medium → standard implementation with tests.

## 1. Goal
Implement end-to-end semantic search: accept a query from the search page, encode it, rank all stored poems by cosine similarity against their stored embeddings, break ties alphabetically by title, label results, and render the top 5 inline on the search page.

---

## 2. Why this task exists now
Task 002 delivered the data pipeline and seeded embeddings. The user cannot yet verify the product's core value — semantic search — until this task lands the ranking and rendering.

---

## 3. Prerequisites
- Task 001: Flask scaffold and search template.
- Task 002: text cleaning, embedding service, repository with poems and stored BLOBs, populated sample data via the CSV import CLI.

## 4. Scope
- `search.py` service module with a `SearchService` that holds a reference to the embedding service and provides `search(query, limit=5)`.
- Relevance labeling constants (thresholds from design doc §6.1.3) in a single constants module (`constants.py`) so they can be adjusted in one place.
- `/search` route (POST) on the public blueprint that reads the form field, cleans / normalizes the query, delegates to `SearchService`, and renders `search.html` with the results list.
- Template updates to `search.html` and a new `_results.html` partial that renders a list of up to 5 result cards with title, preview (first 2 lines of the poem text), relevance label badge, and a `data-poem-id` attribute for the modal wiring in Task 004.
- Query context line `Results for: "..."` above the results.
- Autofocus preserved; the input keeps the submitted value so the user can edit it.
- Empty / whitespace-only query is a no-op (re-render the page, no results region shown).
- Preloaded embedding cache in-memory: on first access, load all `(id, title, embedding)` tuples into a numpy matrix for fast ranking. Invalidate the cache when a poem is added, edited, or deleted — expose a `refresh()` method used in later tasks.

## 10. Classes to Implement or Modify
### SearchService
- Responsibility: Own the in-memory embedding cache, run cosine similarity against it, apply tie-breaking and labeling.
- Key methods: `__init__`, `search`, `refresh`, `_load_cache`.
- Interactions: Reads from `repository.iter_embeddings`; encodes queries via `EmbeddingService`; called by the public route.

### SearchResult
- Responsibility: Immutable value type returned to templates.
- Fields: `id`, `title`, `preview`, `score`, `label`.

---

## 11. Public Functions / Methods
### SearchService.search
- Purpose: Return the top `limit` search results for a cleaned query.
- Parameters: `query: str`, `limit: int = 5`.
- Returns: `list[SearchResult]` of length up to `limit`; empty list when the query is empty or the DB has no poems.
- Side effects: None (read-only; may lazily load the cache).
- Errors: Propagates embedding errors.

### SearchService.refresh
- Purpose: Invalidate and rebuild the in-memory cache.
- Parameters: none.
- Returns: none.
- Side effects: Reads the DB.
- Errors: none.

### public.search
- Purpose: Handle `POST /search`, clean the query, delegate to `SearchService`, render the template with results.
- Parameters: Flask form data.
- Returns: rendered response.
- Side effects: none.
- Errors: converts unexpected service errors to a 500 page (default Flask handling is acceptable for V1).

### repository.iter_embeddings
- Purpose: Yield every poem's `(id, title, cleaned_text, deserialized_embedding)` for cache building.
- Parameters: `conn`.
- Returns: generator of tuples.
- Side effects: reads the DB.

### constants.label_for
- Purpose: Map a cosine similarity score to a label string using fixed thresholds.
- Parameters: `score: float`.
- Returns: `"Strong"`, `"Moderate"`, or `"Weak"`.
- Side effects: none.

## 14. State Transitions
- Before: Search page renders the form; no ranking logic is wired up; `POST /` from Task 001 returned the blank page.
- After: `POST /search` executes the full ranking pipeline and renders labeled results; the cache lazily loads on first search.
- Invariants:
  - When the query is empty or whitespace-only, no search is performed and no results region is rendered.
  - When the query is non-empty and the DB has at least one poem, the response contains `min(5, N)` results.
  - Ties in cosine score are broken by `title ASC`.
  - Every rendered result has exactly one label.

## 15. Detailed Implementation Notes
1. In `constants.py`, define `STRONG_THRESHOLD = 0.45`, `MODERATE_THRESHOLD = 0.30`, and a `label_for(score)` function.
2. `SearchService._load_cache` iterates `repository.iter_embeddings`, stacks vectors into a numpy matrix, and stores parallel metadata. If the DB is empty, the cache holds an empty matrix.
3. `SearchService.search`:
   - If the cleaned query is empty → return `[]`.
   - If cache not loaded → load it.
   - Encode the query using `embedding_service.encode_query(cleaned_query)`.
   - Compute `scores = embeddings @ query_vector` (vectors are already unit-normalized so the dot product equals cosine similarity).
   - Build a list of `(score, title, id)` tuples and sort by `(-score, title)` so ties break alphabetically.
   - Slice the top `limit`, fetch preview text (first two non-empty lines of the cleaned body, with "…" if the body has more lines).
   - Map score → label via `constants.label_for`.
   - Return the `SearchResult` list.
4. Preview computation lives in `search.py` to keep templates simple.
5. Route handler: clean the submitted query using `text_cleaning.clean_query`; if empty, render the home page with no results; else delegate to the search service and render the page with `results`, `query`, and `original_query` (the raw submitted text, for display in the input).
6. `_results.html` renders a `<section class="results">` with `<article class="result" data-poem-id="...">` elements to pave the way for Task 004's modal wiring.
7. `search.html` conditionally includes `_results.html` when `results` is present.
8. CSS adds a badge palette (Strong green, Moderate amber, Weak gray), row hover state, and a narrow-screen rule.
9. The `SearchService` is attached to the Flask app in `create_app`; the public route pulls it via `current_app.extensions["search"]`.

## 16. Acceptance Criteria
1. `POST /search` with `q=grief` renders the page with up to 5 results, each showing title / preview / label.
2. Results are sorted by descending similarity with ties broken alphabetically by title.
3. Label thresholds match design doc §6.1.3: `Strong ≥ 0.45`, `0.30 ≤ Moderate < 0.45`, `Weak < 0.30`.
4. Empty or whitespace-only query re-renders the page with no results region and no query context line.
5. Query context line displays the exact submitted (non-empty) query wrapped in quotes.
6. The search input retains the submitted value after the POST.
7. Results render `Untitled` when the poem title is empty.
8. Tests pass with real SQLite and a real embedding model.

## 19. Risks and Edge Cases
- Risk: Cache becomes stale after admin writes in Tasks 006/007 → addressed when those tasks call `SearchService.refresh`. For Task 003, the only way to change data is the CLI import, which restarts the process.

- Edge case: Query that embeddings poorly → scores are low; all results land under `Weak`. Acceptable per design doc §6.1.2.8.
