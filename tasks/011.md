- Level: Medium

  - Integration complexity: Touches lexical processing, search constants, search service, and public search tests while keeping the UI unchanged.
  - Risk level: Medium because ranking behavior changes globally and must stay deterministic across the full corpus.
  - Estimated effort: ~340 changed lines.
- Complexity rubric alignment:
  - Low: single component, no persistence, minimal side effects
  - Medium: multiple components and/or persistence involved
  - High: cross-cutting concerns, filesystem + DB interaction, concurrency, or irreversible operations

  - Medium → standard implementation with tests

## 1. Goal
Upgrade search from semantic-only ranking to combined semantic + exact lexical ranking using the stored `lemmatized_search_text`, while preserving the simple V1 public UI.

---

## 2. Why this task exists now
After Task 009, lexical data is persisted for every poem, and after Task 010 existing installs are rebuilt safely. The next lowest-risk ranking step is exact lexical matching before adding synonym expansion noise.

---

## 3. Prerequisites
- Task 003: current semantic search pipeline and public results rendering.
- Task 009: persisted `lemmatized_search_text` and shared lexical processor.
- Task 010: automatic upgrade rebuild keeps existing installs current before search uses lexical data.

---

## 4. Scope
- Centralize the V2 scoring constants used by combined ranking and final-score labels.
- Extend lexical processing with query-time tokenization, stopword removal, normalization, and lemmatization for exact lexical matching.
- Extend the in-memory search cache so every cached poem includes its lexical match representation.
- Compute `lexical_score` from exact matches only for this task, averaged across meaningful query words.
- Apply the hard semantic floor and final combined scoring formula across the full poem corpus.
- Base relevance labels on `final_score` while keeping the same visible label text and thresholds.
- Keep the public search page, modal flow, and result count unchanged.

---

## 5. Out of Scope
- WordNet synonym expansion.
- Query-time synonym caching.
- Search diagnostics logging.
- Any new UI controls, explanations, or debug views.

---

## 6. Observable Outcome

- Relevance labels remain `Strong`, `Moderate`, and `Weak`, but now reflect the combined score.
- The public search page still shows exactly 5 results with the same layout and modal behavior.

## 15. Detailed Implementation Notes
Step-by-step plan.
- Expand `constants.py` so all V2 ranking thresholds and weights are defined in one place.
- Extend `LexicalTextProcessor` with a query-term builder that reuses the same normalization and lemmatization logic as stored poem text.
- Cache lexical match data in memory in a form that supports membership checks without frequency counting.
- In `SearchService.search`:
  - build the semantic query exactly as V1 does
  - compute the semantic score for every poem
  - build `Q_terms` from the lexical processor
  - assign `1.0` per query word when the exact normalized term exists in the poem lexical data
  - average the per-word scores across all meaningful query words
  - zero lexical contribution when `semantic_score < 0.20`
  - compute `final_score`
  - sort by `final_score` descending, then title ascending
- Keep the result template unchanged so this task is a pure ranking change, not a UI redesign.
- Preserve the existing top-5 behavior even when the corpus is smaller than 5.

Do not include code.

## 16. Acceptance Criteria
1. Search ranking uses `0.8 * semantic_score + 0.2 * lexical_score` across the full corpus.
2. Exact lexical matching applies to all non-stopword query words and contributes `1.0` per matched word.
3. Multi-word lexical score is the arithmetic average across meaningful query words.
4. Lexical contribution is set to `0.0` when `semantic_score < 0.20`.
5. Relevance labels use the existing thresholds against `final_score`, not semantic score.
6. Search still returns exactly 5 results and keeps the public UI unchanged.
7. Ties are still broken alphabetically by title.

## 18. Failure Handling & Rollback
Required if task modifies or deletes data.

- Failure scenarios:
  - Query lexical processing fails unexpectedly for one search request.
  - Cached lexical data is missing for one row due to DB corruption.
- Partial failure handling:
  - Fall back to `lexical_score = 0.0` for the affected request or row rather than failing the whole search page.
- Rollback strategy:
  - No persistent data rollback is required because this task changes ranking only.
  - If a regression is discovered during implementation, revert to semantic-only behavior by backing out this task, not by introducing runtime toggles here.
- Idempotency considerations:
  - `SearchService.refresh` and repeated searches must remain deterministic for the same corpus and query.

---

## 19. Risks and Edge Cases
- Stopword-only queries produce an empty lexical term list: lexical score must become `0.0`, not divide by zero.
- Empty or whitespace-only queries must preserve V1 no-op behavior.
- Lexical data for title + body must remain equal weight because the stored field combines both without boosts.
- Exact lexical matching can shift labels near thresholds: route and badge tests must be updated deliberately rather than hand-waved.
